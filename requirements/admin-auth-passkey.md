## Feature: Admin Auth — Passkey

Adds passkey (WebAuthn) authentication to the admin console as the primary login method, layered on top of the magic link auth system. Enables the admin to log in with a single biometric/PIN tap instead of waiting for an email. Magic link remains as a fallback for new devices or lost passkeys.

## Out of scope

- **Visual styling of passkey pages** (management page, bootstrap prompt, passkey button on login) — all pages are functional but unstyled; visual design is applied by the Admin Console UI feature
- **Multi-user admin support** — single admin user only; all passkeys belong to the one admin
- **Passwordless fallback beyond magic link** — no SMS, no TOTP, no recovery codes
- **Audit logging of passkey events** (registration, removal, authentication) — potential future enhancement

## Technical Specification

### Summary

Adds passkey (WebAuthn) authentication to the feedmail admin console as a primary login method, layered on top of the existing magic link authentication system. Introduces a new D1 table for passkey credentials, new admin routes for WebAuthn ceremony endpoints and passkey management, and modifications to the existing login flow to offer passkey-based authentication when credentials are registered. The `@simplewebauthn/server` library handles server-side WebAuthn operations; client-side ceremony code is delivered inline via Handlebars templates (no bundler, no `@simplewebauthn/browser` dependency).

### Implementation Notes

**Database migration (R1):** File `migrations/0007_passkey_credentials.sql`. Uses `CREATE TABLE IF NOT EXISTS`. Public key stored as base64url TEXT (D1 BLOB round-tripping unreliable). `transports` column stores JSON array or NULL. `webauthn_challenges` table is ephemeral (5-min TTL). `session_token` links to admin session token (registration) or temporary cookie value (authentication). `type` column discriminates registration vs authentication challenges.

**DB helpers (R2):** Added to `src/admin/lib/db.js` below existing session helpers. `cleanupExpiredChallenges` called fire-and-forget during challenge creation.

**WebAuthn ceremony endpoints (R3-R6):** New file `src/admin/routes/passkeys.js`. RP ID from `env.DOMAIN`. Expected origin `https://${env.DOMAIN}`. Single fixed user ID (one admin). User name from `getCredential(env.DB, "admin_email")`. Challenges in D1 (Workers are stateless). Public key serialized via `isoBase64URL.fromBuffer()`/`.toBuffer()`. JSON request/response using `jsonResponse()`. Authentication challenge cookie: `feedmail_webauthn_challenge` with `HttpOnly; Secure; SameSite=Strict; Path=/admin; Max-Age=300`.

**Login page (R7):** `handleLogin` checks `getPasskeyCredentialCount()`, passes `hasPasskeys` to template. Inline JS (~40 lines): base64url helpers, fetch options, `navigator.credentials.get()`, POST result, handle redirect/error. No `@simplewebauthn/browser`.

**Management page (R8):** New template `admin-passkeys.hbs`. Lists credentials with `{{#each passkeys}}`. Registration form with inline JS. Rename via inline form. Delete with `confirm()`. WebAuthn availability check via `window.PublicKeyCredential`.

**Rename/delete (R9-R10):** Form-encoded POST, redirect to `/admin/passkeys`. Name capped at 100 chars.

**Bootstrap prompt (R11):** Non-persistent dismissal via `?dismissed=passkey` query param. Reappears until passkey registered.

**Routing (R12):** Public auth routes rate-limited as `admin_login`. `getEndpointName` in `rate-limit.js` updated. Dynamic credential ID via path splitting.

**Templates (R13):** `adminPasskeys` registered in `src/shared/lib/templates.js`. Precompile script auto-discovers `.hbs` files.

**Dependency (R14):** `@simplewebauthn/server` ^13.x as production dependency.

**DB reset (R15):** Append `DELETE FROM passkey_credentials; DELETE FROM webauthn_challenges;` to script.

### Implementation Decisions (Out of Spec)

The following decisions were made during implementation and diverge from or extend the original specification:

1. **Rename error on missing credential:** `handlePasskeyRename` returns a redirect with `?error=Passkey not found` when the credential does not exist, rather than silently succeeding. Delete remains idempotent.
2. **Name length rejection:** `handlePasskeyRename` rejects names longer than 100 characters with an error redirect (`?error=Name must be 100 characters or fewer`) rather than silently truncating.
3. **Domain passed to login template:** `handleLogin` in `auth.js` passes `domain: env.DOMAIN` to the `adminLogin` template for constructing absolute WebAuthn API URLs in inline JavaScript.
4. **Shared webauthn-helpers partial:** Base64url conversion functions (`base64urlToBuffer`, `bufferToBase64url`) extracted into `src/templates/partials/webauthn-helpers.hbs` and shared between `admin-login.hbs` and `admin-passkeys.hbs`.
5. **Handler name shortening:** Route handlers use shorter names (`handleRegisterOptions`, `handleRegisterVerify`, `handleAuthenticateOptions`, `handleAuthenticateVerify`) instead of the spec's longer forms (`handleRegistrationOptions`, etc.).
6. **getCookieValue helper:** Generic cookie parser extracted to `src/admin/lib/session.js` and used by both session management and passkey challenge token retrieval.
