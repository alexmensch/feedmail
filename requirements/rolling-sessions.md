## Feature: Rolling Admin Sessions

Admin sessions currently expire at a fixed time after login (24 hours), regardless of activity. This means an admin actively working in the console can be logged out mid-workflow. This feature changes sessions to a rolling expiry model: each authenticated request extends the session, so it only expires after a period of inactivity. An absolute maximum lifetime (7 days from login) forces re-authentication regardless of activity as a defense-in-depth measure.

| --- | ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Rolling session extension     | Each authenticated admin request (any request that passes session validation) extends the session's `expires_at` by 24 hours from the current time. Both the D1 `admin_sessions` row and the session cookie's `Max-Age` are updated. This happens transparently as part of the existing session middleware — no additional requests or user action required. | An authenticated request made at time T updates `expires_at` to T + 24 hours in the database. The `Set-Cookie` header on the response refreshes `Max-Age` to 24 hours. A session that would have expired under the old fixed model remains valid as long as the admin makes at least one request within each 24-hour window.            | Rapid sequential requests each extend the session — this is harmless and correct. The extension must not create a new session row or change the session token.                                                                                                               |
| 2   | Absolute session lifetime cap | A session cannot be extended beyond 7 days after its creation time. The `created_at` column (already present in `admin_sessions`) is compared against the current time on each request. If the session's `created_at` is more than 7 days ago, the session is treated as expired regardless of its `expires_at` value.                                       | A session created at time T is always rejected after T + 7 days, even if `expires_at` has been rolled forward by recent activity. When the remaining time until the absolute cap is less than 24 hours, `expires_at` is set to the absolute cap rather than T + 24 hours (clamped). After absolute expiry, the admin must log in again. | A session with `created_at` exactly 7 days ago is expired. A session at 6 days 23 hours is valid but its next extension is clamped to the 7-day boundary rather than a full 24 hours. The absolute cap is enforced even if the database `expires_at` value is in the future. |
| 3   | Hardcoded constants           | The rolling inactivity timeout (24 hours) and absolute session cap (7 days) are defined as named constants alongside the existing `SESSION_TTL_SECONDS`. These values are not configurable at runtime.                                                                                                                                                       | Constants are defined with clear names that distinguish the inactivity timeout from the absolute maximum. The existing `SESSION_TTL_SECONDS` constant is replaced or repurposed — there is a single, unambiguous set of session lifetime constants.                                                                                     | None — this requirement ensures clarity for future maintainers.                                                                                                                                                                                                              |
| 4   | Session creation unchanged    | New sessions created at login (both magic link and passkey authentication) continue to set an initial `expires_at` of 24 hours from creation. The only change is that this expiry is now rolling rather than fixed. No changes to login flow, session token generation, or cookie attributes (HttpOnly, Secure, SameSite=Strict, Path=/admin).               | Login via magic link creates a session with `expires_at` = now + 24 hours (same as before). Login via passkey creates a session with the same initial expiry. Cookie attributes are unchanged. The session begins its 7-day absolute lifetime window at creation.                                                                       | None — this requirement confirms that login behaviour is unaffected.                                                                                                                                                                                                         |

## Out of scope

- **Configurable timeouts** — both the inactivity timeout and absolute cap are hardcoded constants; runtime configuration via site_config can be added later if needed
- **Expiry warning UI** — no banner or notification when the absolute 7-day cap is approaching; the admin is silently redirected to login
- **Sliding window for the absolute cap** — the 7-day cap is measured from session creation, not from last re-authentication; re-login resets it by creating a new session
- **Per-action re-authentication** — no step-up auth for destructive actions; all authenticated requests are treated equally
