## Feature: First-Time Setup Flow

When a self-hoster deploys feedmail and logs into the admin console for the first time, the dashboard presents a guided setup form for creating their first channel and feed — replacing the empty stats view so the operator can complete instance configuration entirely within the browser.

## Requirements

| # | Requirement | Description | Acceptance Criteria | Edge Cases / Error Conditions |
|---|-------------|-------------|---------------------|-------------------------------|
| 1 | Empty state detection | When the admin console dashboard loads and no channels exist in the database, the dashboard displays the first-time setup form instead of the normal stats view. The detection uses the existing admin API (`GET /api/admin/stats` or equivalent) — not direct D1 access. | Dashboard with zero channels shows the setup form. Dashboard with one or more channels shows normal stats. | Admin deletes all channels after initial setup: the setup form reappears on the next dashboard load. Race condition: channel created by another session between page load and form submission is handled gracefully (API returns success or a clear conflict error). |
| 2 | Channel creation form | The setup form collects all fields needed to create a channel via `POST /api/admin/channels`: channel ID, site name, site URL, from user, from name (required); reply-to, company name, company address (optional); CORS origins; and one feed (name, URL). The form is rendered inline on the dashboard page using the admin console's existing CUBE CSS styling. | All required fields are present and labelled. Optional fields are clearly marked as optional. Form submits to the admin API. Successful submission creates a channel with one feed. | Form submission with missing required fields: inline validation errors shown next to each invalid field. API returns an error (e.g., duplicate channel ID): error displayed clearly without losing form state. |
| 3 | Channel ID validation | The channel ID field enforces a strict slug format: lowercase alphanumeric characters and hyphens only, no leading or trailing hyphens, no consecutive hyphens, no whitespace. Validation feedback is shown inline as the user types or on submission. | Valid IDs accepted: `my-blog`, `newsletter`, `updates-2024`. Invalid IDs rejected with specific guidance: `My Blog` (uppercase/spaces), `-leading` (leading hyphen), `trailing-` (trailing hyphen), `double--hyphen` (consecutive hyphens), `special!chars` (non-alphanumeric). | Empty channel ID: required field error. Very long channel ID: accepted (no max length constraint at this level; API may enforce its own limits). |
| 4 | From user validation | The from user field (email local part) rejects values containing `@` or whitespace, consistent with the existing validation rules. The instance's `DOMAIN` is displayed adjacent to the field (e.g., `@example.com`) so the user sees the full sender address that will be constructed. Inline error message explains the expected format. | `hello` accepted and displayed as `hello@{DOMAIN}`. `hello@domain` rejected with error explaining it should be the local part only (without @). `hello world` rejected. | Empty value: required field error. |
| 5 | CORS origin auto-populate | When the site URL field loses focus and contains a valid URL, the CORS origins field is automatically populated with the derived origin (scheme + host + port). If the user has previously manually edited the CORS origins field, auto-population is disabled — the user's manual value is preserved. | Entering `https://example.com/blog` in site URL and tabbing out populates CORS origins with `https://example.com`. Entering `http://localhost:8888/path` populates `http://localhost:8888`. Manual edit of CORS origins after auto-populate: subsequent site URL changes do not overwrite. | Site URL field empty or invalid on blur: CORS origins field is not modified. Site URL without a scheme (e.g., `example.com`): CORS origins not populated; site URL validation error shown. |
| 6 | Feed fields | The form includes one feed name and one feed URL field. Both are required. Additional feeds can be added from the channel detail page after setup. | Feed name and feed URL fields are present and required. Submission with empty feed name or URL shows inline validation errors. | Feed URL that is not a valid URL format: validation error. Feed name with leading/trailing whitespace: trimmed before submission. |
| 7 | Form submission and transition | On successful channel creation, the dashboard transitions from the setup form to the normal stats view showing data for the newly created channel. The transition does not require a full page reload (HTMX fragment swap). Success feedback is shown confirming the channel was created. | Successful submission: setup form is replaced by the normal dashboard content. Success message is visible. Stats reflect the new channel (zero subscribers, zero sent items). | Network failure during submission: error message displayed without losing form state. Slow API response: loading indicator visible during submission. |
| 8 | Contextual guidance | Form fields include brief helper text explaining their purpose and expected format, oriented toward a self-hoster who may be configuring feedmail for the first time. Helper text is concise — not tutorial-length. | Each field has associated helper text or placeholder that clarifies its purpose (e.g., channel ID: "unique identifier used in your subscribe form"; from user: "email local part — combined with your domain to form the sender address"; site URL: "your website's address, used for linking back in emails"). | Helper text does not interfere with inline validation error messages — both can be visible simultaneously or errors replace helper text contextually. |

## Out of scope

- **Multi-step wizard or separate setup route** — the setup form is a single inline form on the dashboard, not a `/admin/setup` wizard
- **Multiple feeds during initial setup** — one feed only; additional feeds are added from the channel detail page
- **Skipping setup** — the form is the dashboard empty state; there is no way to dismiss it without creating a channel (the normal dashboard requires channel data to display)
- **Draft persistence** — navigating away from the form loses all entered data; no auto-save or local storage
- **Setup progress indicator** — there is no multi-step progress bar since it's a single form
- **Onboarding tour or walkthrough of other admin pages** — this feature covers channel creation only, not a general admin onboarding experience
- **Subscribe form code generation** — the setup form does not generate embed code or integration instructions for the operator's website (potential future enhancement)
