## Feature: Channel Form Improvements

The channel creation and editing form is improved to combine channel configuration and feed management into a single page, add inline validation with clear error messages, and provide contextual helper text for each field — so that operators can configure a channel and its feeds in one place without navigating between multiple pages.

## Requirements

| # | Requirement | Description | Acceptance Criteria | Edge Cases / Error Conditions |
|---|-------------|-------------|---------------------|-------------------------------|
| 1 | Unified channel and feed form | The channel form (both create and edit modes) displays channel configuration fields and all associated feeds on a single page. Channel fields and feed rows are saved together via a single "Save" action. | Creating a channel: channel fields and at least one feed are submitted together. Editing a channel: channel fields and all feed changes (additions, edits, deletions) are submitted together. No separate pages required for feed management. | Channel creation with zero feeds: form requires at least one feed row to submit. JavaScript disabled: form still functions (feed rows are static HTML, no dynamic add/remove). |
| 2 | Inline feed management | The form displays feeds as editable rows. Each row has fields for feed name and feed URL. Users can add new feed rows and remove existing ones without leaving the page. Removing a feed row shows a confirmation prompt before the row is removed from the form. | Add feed: clicking an "Add feed" control appends a new empty row. Remove feed: clicking remove on a row prompts for confirmation, then removes the row from the form. Edit feed: existing feed rows are editable inline. Changes are not persisted until the user saves the form. | Removing the last feed row: blocked — at least one feed is required. Adding a feed row with a duplicate name or URL of another row: caught on submission (server-side validation), not inline. Feed rows added then removed before saving: no API calls made for those rows. |
| 3 | Channel ID slug validation | The channel ID field enforces a strict slug format: lowercase alphanumeric characters and hyphens only, no leading or trailing hyphens, no consecutive hyphens, no whitespace. Validation feedback is shown inline on the field. This validation applies on both the create and edit forms. | Valid IDs accepted: `my-blog`, `newsletter`, `updates-2024`. Invalid IDs rejected with specific guidance: `My Blog` (uppercase/spaces), `-leading` (leading hyphen), `trailing-` (trailing hyphen), `double--hyphen` (consecutive hyphens), `special!chars` (non-alphanumeric). | Empty channel ID: required field error. Channel ID field on edit form: displayed but not editable (channel IDs are immutable after creation). |
| 4 | From-user validation | The from-user field (email local part) rejects values containing `@` or whitespace. The instance's DOMAIN is displayed adjacent to the field so the user sees the full sender address that will be constructed (e.g., `@example.com`). Inline error message explains the expected format. This validation applies on both create and edit forms. | `hello` accepted and displayed as `hello@{DOMAIN}`. `hello@domain` rejected with error explaining it should be the local part only. `hello world` rejected. | Empty value: required field error. |
| 5 | CORS origin auto-populate | On the channel creation form only, when the site URL field loses focus and contains a valid URL, the CORS origins field is automatically populated with the derived origin (scheme + host + port). If the user has previously manually edited the CORS origins field, auto-population is skipped — the user's manual value is preserved. | Entering `https://example.com/blog` in site URL and tabbing out populates CORS origins with `https://example.com`. Entering `http://localhost:8888/path` populates `http://localhost:8888`. Manual edit of CORS origins after auto-populate: subsequent site URL changes do not overwrite. | Site URL field empty or invalid on blur: CORS origins field is not modified. Site URL without a scheme (e.g., `example.com`): CORS origins not populated. Edit form: auto-populate is disabled regardless of user interaction — existing CORS origins are always preserved. |
| 6 | Contextual helper text | Form fields include brief helper text explaining their purpose and expected format, oriented toward an operator who may be configuring feedmail for the first time. Helper text is concise. This applies to both create and edit forms. | Each field has associated helper text that clarifies its purpose. Examples: channel ID — "unique identifier used in your subscribe form"; from user — "email local part, combined with your domain to form the sender address"; site URL — "your website's address, used for linking back in emails"; CORS origins — "origins allowed to submit the subscribe form". | Helper text does not interfere with inline validation error messages — both can be visible simultaneously, or errors replace helper text contextually. |
| 7 | Form error handling | API errors during save are displayed clearly without losing form state. Validation errors from the API (e.g., duplicate channel ID, duplicate feed name/URL) are shown to the user with enough context to correct the problem. | API validation error: error message displayed, all form fields retain their values. Network failure: error message displayed, form state preserved. Partial failure during multi-call save (e.g., channel saved but a feed creation fails): error indicates which operation failed. | Concurrent edit: if another session modifies the channel between load and save, the API error is surfaced clearly. |
| 8 | Dashboard empty state link | When no channels exist, the dashboard's empty state message links directly to the channel creation form (`/admin/channels/new`) instead of the channel list page. | Clicking the empty state link navigates to the channel creation form in one step. | If channels are created by another session, the link still works — the user simply creates an additional channel. |

## Out of scope

- **Dashboard inline form** — the dashboard empty state is updated to link directly to the channel creation form, but the form itself is not embedded on the dashboard; no HTMX swap
- **Client-side URL or feed validation** — feed URLs and site URLs are validated server-side by the API; the form does not validate URL format or reachability client-side (beyond the CORS auto-populate needing a parseable URL)
- **Drag-and-drop feed reordering** — feeds are displayed in their natural order; no reordering capability
- **Bulk feed import** — feeds are added one at a time via the inline rows
- **Auto-save or draft persistence** — navigating away from the form loses unsaved changes
