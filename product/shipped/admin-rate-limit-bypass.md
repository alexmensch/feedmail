## Feature: Admin Rate Limit Bypass for Internal Service Binding Requests

Authenticated admin console usage should not be subject to API Worker rate limiting. Requests from the Admin Worker via the Cloudflare Service Binding are legitimate, session-authenticated traffic that currently accumulates rate limit entries under IP `"unknown"` and can block an active admin user. Internal requests should be identified explicitly and exempted from rate limiting when accompanied by a valid API key.

## Out of scope

- **Cleaning up existing `"unknown"` rows in `rate_limits`**: Existing rows from past service binding calls will be pruned naturally by the probabilistic 7-day cleanup.
- **Rate limiting changes on the Admin Worker**: The Admin Worker's own rate limiting of public routes (`admin_login`, `admin_verify`) is unrelated and unchanged.
- **Session-awareness in the API Worker**: The API Worker does not need to know about admin sessions. The internal header + API key is sufficient to identify legitimate internal traffic.
- **Configurable bypass behaviour**: The bypass is unconditional for authenticated internal requests. There is no need for a toggle or separate rate limit tier.

## Technical Specification

### Summary

Authenticated requests from the Admin Worker to the API Worker via the Cloudflare Service Binding are currently rate-limited under IP `"unknown"`, which can block legitimate admin console usage. This feature adds a custom header (`X-Internal-Request`) to all `callApi()` requests and modifies the API Worker's request pipeline to defer rate limiting for `/api/admin/*` requests bearing that header until after authentication. Authenticated internal requests skip rate limiting entirely; failed-auth internal requests are rate-limited retroactively.

### Implementation Notes

**File changes:**

- `src/admin/lib/api.js` — Add `"X-Internal-Request": "true"` to the headers in `callApi()`
- `src/api/worker.js` — Add `INTERNAL_REQUEST_HEADER` constant, `isInternalRequest(request)` helper, and conditional deferral logic in the rate limiting block. For internal admin requests: auth first, then skip rate limiting on success or apply it retroactively on failure.

**No changes needed to:** `src/shared/lib/rate-limit.js`, `src/shared/lib/config.js`, `src/api/routes/admin.js`, `src/admin/worker.js`

**Test files:** `tests/admin/lib/api.test.js` (verify header present), `tests/api/worker.test.js` (all 5 requirement scenarios)

### Out of Scope

- Cleaning up existing `"unknown"` rows — pruned naturally by 7-day cleanup
- Admin Worker's own rate limiting of public routes — unrelated
- Session-awareness in the API Worker — internal header + API key is sufficient
- Configurable bypass behaviour — no toggle needed
- Rate limit bypass for `/api/send` — header ignored for non-admin endpoints
- Stripping `X-Internal-Request` header — harmless if it reaches route handlers; external callers gain no advantage
